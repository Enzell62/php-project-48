// =>     [proxy] => 123.234.53.22
// =>     [follow] =>
// => )
Функция возвращает просто объект stdClass, который похож на ассоциативный массив:
<?php
foreach($data as $key => $value) {
  var_dump("Key: {$key}, value: {$value}");
}

// => Key: host, value: hexlet.io
// => Key: timeout, value: 50
// => Key: proxy, value: 123.234.53.22
// => Key: follow, value: false
Теперь можем например получить ключи:

И получить значение, используя ключ:

var_dump($data->$key); // => hexlet.io

Сравнение плоских файлов (JSON)
Дифф строится на основе того, как файлы изменились относительно друг друга, ключи выводятся в алфавитном порядке. Ниже пример того, что должно получиться по результату данного шага:
gendiff filepath1.json filepath2.json

{
  - follow: false
    host: hexlet.io
  - proxy: 123.234.53.22
  - timeout: 50
  + timeout: 20
  + verbose: true
}
Отсутствие плюса или минуса говорит, что ключ есть в обоих файлах, и его значения совпадают. Во всех остальных ситуациях значение по ключу либо отличается, либо ключ есть только одном файле. В примере выше ключ timeout есть в обоих файлах, но имеет разные значения, proxy находится только в file1, а verbose только в file2.
Любая программа, распространяемая в виде пакетов, одновременно является и библиотекой. Другими словами, библиотеку которую мы разрабатываем, можно установить как зависимость в любом другом пакете. Пример того, как другие пакеты могли бы использовать наш gendiff:
<?php

use function Differ\Differ\genDiff;

$diff = genDiff($pathToFile1, $pathToFile2);
print_r($diff);
То есть ваша библиотека должна предоставлять функцию, вызов которой возвращает разницу между данными в виде строки.

•	Сравниваются данные, а не строки файлов.
•	Две строчки диффа, отвечающие за различия поля, должны находиться рядом. Причем в начале должна выводиться строка относящаяся к первому файлу, а затем строка относящаяся ко второму файлу (см. пример с timeout).

Есть утилиты, которые также сравнивают и выводят разницу между двумя текстовыми файлами — например, diff. Формат вывода может отличаться, но смысл остается: добавленные строки обозначаются одним знаком, удаленные — другим. Оставшиеся без изменения либо опускаются, либо отбиваются пробелом.
Это сделано на этапе сравнения плоских файлов. Теперь нужно сделать аналогично для файлов с вложенными структурами. Обозначения остаются аналогичными, но добавляется сложность в подсчете отступов у добавленных знаков.
Если изучить диф выше, можно увидеть закономерности: каждое погружение на один уровень добавляет четыре знака. Они могут состоять либо только из пробелов, либо из двух пробелов, спецсимвола и еще одного пробела.
Можно пойти дальше и представить, что у вложенного типа и у неизмененного есть свой спецсимвол — пробел.
Чтобы вычислить отступ, можно использовать формулу глубина * количество отступов — смещение влево. В этом случае смещение влево — это отступ в обратную сторону, то есть специальный символ + пробел между ним и строкой.
Например, для ключа ops в дифе выше количество считается так: уровень вложенности — 3, количество пробелов на один уровень — 4, смещение влево — 2. В итоге получается 10 пробелов до спецсимвола.
Будет нагляднее, если заменить пробелы на точки, а спецсимволы и пробел после них оставить без изменений. В этом примере хорошо видно, как работает формула смещения влево:
{
..  common: {
......+ follow: false
......  setting1: Value 1
......+ setting5: {
............key5: value5
........}
....}
}
Также в глаза бросается строка key5: value5, перед ней нет никаких спецсимволов и дополнительного смещения влево. Это происходит из-за того, что это значение, приведённое к строке, а не отдельная вложенная структура.
